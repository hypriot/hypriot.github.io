Creating the Docker Engine for a high-density ARMv8 server from HiSilicon

Mon, Feb 29, 2016

Running Docker containers on the latest high-density ARMv8 cloud servers lets you operate your data center at lower costs and gives you the ability to scale even faster. In this article we demonstrate the technical ground work, how to compile the Docker Engine on a brand-new hardware platform and bootstrap all the development tools we need for this purpose.

Introduction

We at Hypriot are working intensively to bring the Docker container technology to life on almost all major 32-bit ARMv6 and ARMv7 platforms. This journey started almost 15 months ago, when we tried to use Docker the first time on a Raspberry Pi and leads to the first HypriotOS release in February 2015. A Debian-based Linux operating system which is ready-to-use and comes with all the latest Docker tools already pre-installed. Right now we are supporting Docker on a fast growing number of devices, like the Raspberry Pi (all models), ODROID C1+ and XU4, and so on. It turns out that it is very easy and efficient to run Docker containers on the smallest ARM and IoT devices which can run at least a basic Linux operating system.

With the experience and expertise of running Docker containers on resource limited devices it’s easier for scaling up for all cloud scale use cases. In the meantime I also did some research how to run Docker even on the latest ARMv8 processors, but this at a time when I don’t get access to real 64-bit ARMv8 hardware. In this case I had to work with QEMU to emulate an ARMv8 system and created and published this work as a public Github repo. So everybody was able to spin up a Droplet at DigitalOcean and could start a 64-bit ARMv8 machine to compile the Docker Engine manually within the emulated QEMU environment.

And luckily from time to time I’m getting some questions about if I could assist to create and installing the latest Docker Engine on a new ARMv8 high-density server. This happens at the end of last year as Shameerali Kolothum Thodi (or Shameer for short) from the company HiSilicon contacted me. Finally we agreed on a collaboration and I got remote access to one of their brand-new development servers, a 16-core ARMv8 Cortex-A57 machine.

HiSilicon is a subsidiary of Huawei and they are developing a new generation of ARMv8 server which is able to run a larger number of power-efficient ARMv8 Cores to build high-density cloud servers. With offices in Cambridge, UK they are located in the right place and have direct contact into the latest ARM research.

Management Summary

Today all the major cloud providers are using Intel-based server systems to run the cloud infrastructure for their customers and their own requirements. With the ever faster growing demand on cloud machines, they have to keep up with the fast pace and installing more and more servers and building up new data centers to have enough space for new racks to get more computing power. With this strong demand the biggest cloud providers suffering from two major problems. First is the computing power of the current CPU’s and second and the power and cooling needed to operate these servers. The important metric here is to have a high-density of computing power per Watt usage. If they were able to run the same workload with smaller servers, they could pack more servers into a single rack and more servers into a given data center. If these smaller servers would also consume less electrical power then they would need less cooling and the systems could be run more efficiently - that’s exactly the goal for a Green Data Center as well. Smaller and more power efficient server systems leads to lower operating costs for building the data centers and for the monthly bill of the electrical power they consume.

Here comes the new ARMv8 server technology into the game. ARM processors are small, powerful and can be run with a high power-efficiency. ARM is working hard to get these visions into reality and they announced the firsts specs of an ARMv8 server CPU already a few years ago. After a few delays, it seems that this technology is finally available for large-scale usage and we’re currently seeing more and more manufacturers with ready-to-use high-end ARMv8 servers are hitting the market. Right now even the largest cloud providers are currently running secret proof-of-concepts in their labs to battle-test these new ARMv8 high-density servers and how they compete against the actual hardware they’re using in their current data centers.

When we’re talking about the high-density of the server hardware itself, we could also talk about high-density of the software stack and workload. And with the help of Docker containers we’re able to increase the density of the software stack to a newer level too. Then we’re able to run a higher workload, or a higher number of applications on the same hardware as before. So one could say, combining these both technologies of high-density ARMv8 servers and high-density software stacks running on Docker containers are the foundation of a new revolution in the cloud data centers of the very near future.

In a recent article ARM Servers: Throwing Down The 25 Percent Share Gauntlet the author stated “PayPal’s math showed that the X86 iron would cost 1.8X as much as the” …ARMv8… “machines, would consume 7X the power, and offer one-tenth the node density per rack. PayPal did not say how much cost savings – if any – came from the move to the” …ARMv8… “chips, but did say there was a “game changing” cost per watt per cubic foot benefit over the X86 machinery.”

And in the end this new technologies cuts the operating costs of your data centers and gives you the velocity to increase your cloud capacity faster then you can deploy new hardware servers.

b) Installing and compiling the latest Go version

To compile the Docker Engine we need the very latest release version of Go, which is currently Go 1.6. You can easily check all available Go versions at The Go Programming Language - Downloads.

Since the release 1.5 of the Go language, the Go compiler is written in pure Go source code and can be compiled itself with an earlier Go version - and this is the recommended way. As we are running an almost current Ubuntu version, we’re looking for the Go version which is available for Ubuntu 15.04 and installing it for now.

# apt-get update
# apt-get install -y golang-go

# go version
go version go1.4.2 gccgo (Ubuntu 5.1~rc1-0ubuntu1.1) 5.0.1 20150414 (prerelease) [gcc-5-branch revision 222102] linux/arm64

With this Go 1.4.2 version provided by Ubuntu 15.04 we’re at least able to bootstrap to the current Go 1.6 release version. So let’s proceed and bootstrap Go 1.6 and get the install path of the Go binary which we need later on.

# which go
/usr/bin/go

We need the path /usr where the Go compiler /bin/go is located to start the bootstrap process.

Next we are downloading the source code of the Go compiler and extracting it to the default location /usr/local/go where we can use the Go compiler later on. Then start the compile script and wait some time.

# export GO_VERSION=1.6
# curl -sSL "https://storage.googleapis.com/golang/go${GO_VERSION}.src.tar.gz" | tar xz -C /usr/local
# cd /usr/local/go/src
# time GOROOT_BOOTSTRAP=/usr ./make.bash

This is the first time, we can get an idea of the performance of this 16-core ARMv8 machine.

Installed Go for linux/arm64 in /usr/local/go
Installed commands in /usr/local/go/bin

real	3m19.853s
user	5m22.660s
sys	9m15.532s

Compiling the complete Go compiler within 9 minutes only - that’s really fast!

Now we do have Go 1.6 installed and it’s almost ready to use. The only thing we have to do now is, including the Go compiler binary to the standard PATH environment variable. This can be done by including this code snippet to our root users ~/.profile script.

# set PATH so it includes GO bin if it exists
if [ -d "/usr/local/go/bin" ] ; then
  PATH="/usr/local/go/bin:$PATH"
fi

After the next login, we should have enabled Go 1.6 by default.

# go version
go version go1.6 linux/arm64

# which go
/usr/local/go/bin/go

Just to avoid any conflicts with the old Go 1.4.2 version we installed for bootstrapping, let’s remove all these packages and clean up the system. For this purpose we check which golang packages were installed previously.

# dpkg -l | grep golang
ii  golang-go                        2:1.3.3-1ubuntu4                  arm64        Go programming language compiler
ii  golang-src                       2:1.3.3-1ubuntu4                  arm64        Go programming language compiler - source files

And now let’s remove them completely and double-ckeck if there is nothing left.

# apt-get --purge remove -y golang-go golang-src
# dpkg -l | grep golang

Results: With some easy commands we were able to installed and compile the latest Go compiler release v1.6 and it’s now ready to use.

c) Bootstrapping the Docker Engine the first time

In order to compile the Docker Engine directly on our development board, we have to know the different methods, how we can and how we should compile the Docker Engine.

First of all, the official process to compile Docker is to run the complete build process within a Docker container. But without an already running Docker Engine on our dev machine, we cannot do this - that’s clear, right?

For this purpose there is an easy, but not really well-known workaround. We have to check and install the necessary development dependencies first and then we can run the build script natively to get a first working Docker binary. So, let’s just do it right away.

With the previous step we’ve already installed the most important dependency with the Go 1.6 compiler, but we’ll need something more.

# apt-get install -y git-core
# apt-get install -y btrfs-tools libsqlite3-dev libdevmapper-dev

After installing these packages it’s a good practice to check and document all the installed versions.

# dpkg -l | grep -E "btrfs-tools|libsqlite3-dev|libdevmapper-d
ii  btrfs-tools                      3.17-1.1                          arm64        Checksumming Copy on Write Filesystem utilities
ii  libdevmapper-dev:arm64           2:1.02.90-2ubuntu1                arm64        Linux Kernel Device Mapper header files
ii  libsqlite3-dev:arm64             3.8.7.4-1ubuntu0.1                arm64        SQLite 3 development files

If you have already installed these packages previously, then it could also be a good idea to upgrade them to the latest version.

# apt-get upgrade -y btrfs-tools libsqlite3-dev libdevmapper-dev

With hopefully all the necessary build dependencies installed now on our machine, we can clone the Docker GitHub repo and compile Docker from it’s source code. As we know, the latest Docker release is v1.10.2 we’re checking out this official release.

# mkdir -p ~/code
# cd ~/code
# git clone https://github.com/docker/docker
# cd docker
# git checkout v1.10.2

And here comes the magic trick to compile Docker natively on our host machine without the need of having a Docker Engine already running. This build will take a few minutes, and as long as we have all the necessary dependencies installed it should succeed too.

# time AUTO_GOPATH=1 ./hack/make.sh dynbinary

# WARNING! I don't seem to be running in the Docker container.
# The result of this command might be an incorrect build, and will not be
#   officially supported.
#
# Try this instead: make all
#

# cd ~/code/docker/

# cp bundles/1.10.2/dynbinary/docker-1.10.2 /usr/bin/docker
# cp bundles/1.10.2/dynbinary/dockerinit-1.10.2 /usr/lib/docker/dockerinit

e) Compiling the Docker Engine the Official Way

With all the details and steps we have done so far, we’re now having a running Docker Engine of the latest release version and can start creating Docker the official way. This recommended way is compiling Docker within a Docker container itself.

First checkout the Docker repo and use the current development version from the master branch. This is necessary because the build tools for ARMv8 are only available in latest master.

# cd ~/code/docker
# git pull
# git checkout master

And install some more build dependencies, if they are not already installed.

# apt-get install -y make

To build the builder Docker Image, we only have to run make build. But please be patient as this will take a really long time.

# make build

The build is starting with the following line, indicating that there will be a Docker Images build with the ARMv8 specific Dockerfile.aarch64.

docker build  -t "docker-dev:master" -f "Dockerfile.aarch64" .

As the result the Docker Image docker-dev is built, which will be used for all the subsequent Docker builds.

Conclusion

As you can see, the latest release of the Docker Engine v1.10.2 is running on this ARMv8 server board pretty well. But right now the ARMv8 architecture is not officially supported by Docker, although the builds are working and at least the basic support is already merged into the master branch.

In order to get a fully supported Docker release for ARMv8 there has to be done a lot more groundwork and improvements in the coming weeks and months. All the unit tests and integrations tests should be fixed and successfully tested on ARMv8. The Docker CI pipeline has to be expanded with a number of ARMv8 build servers, so that all new changes and pull request will be checked and verified automatically on real ARMv8 hardware too.

With this technical walkthrough you can clearly imagine that the Docker container technology is almost ready-to-use on the new generation of high-density ARMv8 cloud servers.
